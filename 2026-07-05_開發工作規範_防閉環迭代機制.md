# 開發工作規範——防閉環迭代機制

> 適用對象：所有在本專案工作的 Agent（小航／Codex，不論底層模型）。
> 定位：CLAUDE.md／AGENTS.md 之下的**執行層規範**，把過去踩坑換來的判斷力固化成規則。與 memory.md 條目衝突時以 memory.md 為準（它記的是定案）。
> 本檔由 memory.md 的索引條目指到，開工必讀順序：CLAUDE.md（或 AGENTS.md）→ status.md → log.md 近 3 筆 → memory.md → **本檔** → 當前工作包的規劃書章節。

---

## 一、任務卡先行（沒有驗收標準不准動手）

動任何程式前，先在回報訊息裡寫出這四行（不用另開檔案）：

```
目標：<一句話，玩家可感知的行為變化>
驗收：<可觀察、可操作的判定條件，至少 2 條>
影響檔案：<預計要動的檔案清單>
風險：<會不會動存檔？會不會動共用元件？>
```

寫不出「驗收」＝還沒想清楚 → 回去讀規劃書／建構書，或問老闆。
規劃書（`2026-07-05_遊戲完善總體規劃書.md`）已為每個工作包寫好驗收標準，直接引用即可。

## 二、先重現、再修（bug 鐵則）

- 修 bug 前必須**先重現**：寫下具體重現步驟與觀察到的錯誤值（畫面文字、console、變數值）。
- 重現不了 → 不准「猜著修」。記錄嘗試過程到 log.md，標「無法重現」，回報老闆補資訊（如老闆截圖是舊版快取——2026-07-02 阿迪卡案例）。
- 修完必須用同一組重現步驟驗證「症狀消失」，並附證據（實測數值／截圖說明）。

## 三、迭代上限（防閉環核心）

- **同一個問題，同一種方法，最多嘗試 2 次。** 第 2 次仍失敗：
  1. 停手。把「症狀／嘗試過什麼／結果／目前假設」寫進 log.md。
  2. 換一種方法（不同切入點，不是微調參數再來一次），或升級處理：問老闆／交接另一個 Agent（交接時 log 必附上面那份紀錄）。
- **禁止第三次相同嘗試。** 判斷「相同」的標準：如果這次修改失敗的理由和上次一樣，它就是相同嘗試。
- 警訊自查：連續 3 輪「改→測→不對→再改」而每輪只動幾個數字 → 你已在閉環裡，停。

## 四、視覺對位禁令（安海之痛，不再重演）

- **禁止**用「截圖→目測→改座標→再截圖」迴圈做素材對位，超過 2 輪即違規。
- 對位一律走工具化流程：
  - 港町 layout：用 `tools/calibrate-port-town.html`（WP-2 建置）描點匯出 JSON，再跑 validate。
  - 世界地圖／探索點：用既有 `tools/validate-m5-2-6-v3-geography-runtime.py` 與 overlay preview 圖。
  - 沒有現成工具 → **先花時間寫產生 overlay 疊圖的小工具**，再對位。寫工具的半小時永遠比目測迭代便宜。
- 新素材生成採「版型先行」：先定 layout 資料，prompt 指定構圖去配合資料；不合就重抽圖，不遷就圖改資料（詳規劃書 WP-2C）。

## 五、資料驅動鐵則

- 場景程式裡**禁止**出現 `port.id === 'xxx'`、`mate.id === 'yyy'` 這類個體特判。需要個體差異＝在 JSON 資料加欄位。
- 遊戲內容（文本、座標、數值、條件）一律進 `src/data/`；程式只管機制。
- 已知既有違規：PortScene 的安海硬編碼（WP-2 段落①負責清除）。清除前不要模仿它新增第二例。

## 六、驗證腳本優先

- 改資料檔必跑對應驗證工具，全綠才 commit：
  | 資料 | 工具 |
  |------|------|
  | mates.json | `node tools/validate-mates-data.mjs` |
  | story.json（WP-1 後） | `node tools/validate-story-data.mjs` |
  | portTownLayouts.json（WP-2 後） | `node tools/validate-port-layouts.mjs` |
  | 地圖／探索座標 | `python tools/validate-m5-2-6-v3-geography-runtime.py` |
  | codex 來源（story MD ✦／discoveries／mates） | 重跑 `node tools/generate-codex-data.mjs`（**勿直改 codex.json**，會被覆蓋） |
- 新增一種資料結構＝同一段落交付它的 validate 工具（照抄 validate-mates-data.mjs 的骨架）。

## 七、存檔升版 checklist（動 `state.ts` 存檔欄位時逐項打勾）

1. `SAVE_VERSION` +1；2. 升版鏈尾端加遷移函式（舊欄位補預設值，參考 v14～v18 前例）；3. 用**上一版實際存檔**測載入（不是新開局）；4. 存檔會被舊版程式讀壞的風險寫進回報；5. status.md 註記新版本號。

## 八、技術紅線速查（詳情見 memory.md 對應日期條目）

- **排版**：一律用 `BASE_W/BASE_H`（1280×720），嚴禁 `this.scale.width/height`（2× 超取樣後是 2560）。新場景不用自己設相機，main.ts hook 會套。〔2026-06-25〕
- **相機跟隨**：繼續用 `startFollow`＋`setBounds`，hook 自動接管；不要自寫捲動。〔2026-06-25〕
- **文字**：一律 `textStyle()`；禁止動 `mipmapFilter`（踩過雷，會更糊）。〔2026-06-25〕
- **測試 port**：固定 `127.0.0.1:5173`（strictPort）；別的 port 存檔不共用，不得用來驗收。不覆蓋未知存檔格。〔2026-06-20〕
- **素材載入**：走 `src/art.ts` 的 `import.meta.glob`＋BootScene preload＋`textures.exists()` fallback；不搬 public、不寫絕對路徑。〔2026-06-19〕
- **素材切片**：imagegen 圖板禁止平均等分切，必須元件偵測（各切片腳本已內建；重繪 source 先看 contact sheet）。〔2026-06-21/22/23〕
- **探索/風景座標**：互動判定用 `approachX/approachY`，圖示用 `x/y`；新增點必跑可航行性驗證。〔2026-07-01〕
- **版權**：只用 CC0/CC-BY 或自製；加素材當下登錄 `assets/CREDITS.md`；CC-BY 要遊戲內標註（每位作者一行）。
- **賣價**：一律走 `sellPriceOf()`，勿在場景重算。〔2026-06-25〕
- **log/memory**：prepend-only、log 必標操作者；流水帳進 log 不進 memory。

## 九、段落交付與回報

- 段落＝可獨立驗證的最小單位。每段落收尾五件套：**build 通過 → 5173 實測 → commit+push（繁中訊息）→ log.md → status.md 勾選**。
- 回報老闆格式：先講「現在遊戲能做什麼了／玩起來差在哪」，再講技術（可省略）。卡住時講「卡在哪、試過什麼、需要老闆做什麼」。
- 準備給老闆試玩時：明確寫「請在 5173（或正式網址）測 X、Y、Z 三件事」。
- 驗證跟隨／動畫類功能時注意：preview 分頁未聚焦 RAF 會暫停，用 `window.__game.step(t,16)` 手動驅動幀。〔2026-06-25〕

## 十、範圍紀律

- 一次只開一個工作包（規劃書 §〇 順序）；工作包內發現的**新問題**不順手修——記到 status.md 待辦，除非它擋住當前驗收。
- 「順便重構」禁止。重構要嘛是工作包明定內容（如 WP-2 段落①），要嘛另立任務卡。
- 建構書已定的設計要改 → 先提案問老闆（CLAUDE.md 既有規則，重申）。
