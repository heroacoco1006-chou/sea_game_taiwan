# M5-8 網頁版架構優化規劃

> **2026-07-07 小航覆核修訂**：①§3-1／3-2 載入架構改為與已完成的 M5-8b 實作一致——**正式採「各場景 preload() 分散載入」**，不做集中式 AssetManager，M5-8d 只補進度顯示與失敗重試，禁止重構已驗證的載入路徑。②新增 M5-8d-2 圖檔壓縮（總體規劃書 §七-3 原定項目，Codex 版遺漏）。③M5-8f 補 GitHub Pages 部署具體指引。④新增「八、技術約定與踩雷區」——執行本規劃前必讀，違反其中任何一條都會破壞既有已驗證架構。

## 一、目的與邊界

本階段讓 GitHub Pages 網頁版在桌面、iPad、iPhone 與 Android 瀏覽器具備可接受的首次載入、觸控、畫面比例與效能。玩法、存檔資料結構與歷史內容不在本階段變更範圍；M5-7 新手教學依老闆決策維持暫緩。

M5-8 必須解決四件事：

1. 首次啟動不再等所有美術素材下載後才顯示標題。
2. 場景需要的素材按群組載入，且下載期間有明確進度與失敗提示。
3. 行動裝置依 GPU／螢幕條件選擇合理超取樣倍率，避免固定 2× 造成 4 倍填充率。
4. 觸控、橫直向、動態視窗高度與瀏海安全區有明確行為與驗收矩陣。

## 二、現況基線（2026-07-06）

### 2-1. Production build

- Vite production build：405 modules。
- `dist/`：354 個檔案，共 160.88 MB。
- 主程式 chunk：1.95 MB（gzip 507 KB）。
- 章節專屬劇情背景：30 張，共 41.25 MB。
- 三主角通用劇情背景：3 張，共 3.85 MB。
- 圖鑑插圖：120 張，共 11.23 MB。
- BGM：7 首，共 41.25 MB；目前由 `AudioManager` 在實際播放曲目時 `fetch()`，不是 BootScene 阻塞項。
- 港町精緻底圖：4 張，共 17.23 MB；目前全部在 BootScene 載入。

### 2-2. 啟動鏈問題

`art.ts` 先以 eager glob 建立所有 URL，`BootScene.preload()` 再把角色、船、世界、全部港町、全部劇情背景、全部章節背景與 120 張圖鑑插圖一次交給 Phaser Loader。eager URL 本身不是主要傳輸問題；真正阻塞標題畫面的是 BootScene 主動下載全部圖檔。

本機 dev 重載約十多秒才從純色畫面進入標題；這只作開發基線，不代替 production／行動網路量測。

### 2-3. 顯示與輸入問題

- 遊戲邏輯尺寸固定 1280×720，canvas 固定 2560×1440，`Scale.FIT` 再縮放到視窗。
- 固定 `SS=2` 在手機同樣承擔 4 倍像素填充率。
- `#game` 使用 `100vh`，未處理 iOS 動態工具列的 `100dvh`、safe-area inset、overscroll 與 touch-action。（**此項已於 2026-07-07 M5-8c 修正**，M5-8e 只做驗收，勿重複實作。）
- 港町已有點擊移動，多數按鈕已有 pointer 操作；世界地圖核心航行仍要逐項確認純觸控是否完整，不可只驗「按鈕能點」。

## 三、目標架構

### 3-1. 資產分層

| 群組 | 內容 | 載入時機 |
|---|---|---|
| `boot-core` | 標題背景、程式生成貼圖、標題 UI 必要內容 | 首次啟動，完成後立即進標題 |
| `navigation-core` | 當前主角行走圖、世界地圖正式圖、世界船隻、必要 HUD／港口圖示 | 新局或讀檔確認後、進世界地圖前 |
| `port:<theme>` | 當前文化圈港町底圖、設施與建築圖 | 進該文化圈港町前；同主題只載一次 |
| `ships` | 船卡、戰鬥船、裝備外觀 | 首次進造船廠、船隊頁或戰鬥前 |
| `story:<hero>:<chapter>` | 當章專屬背景與必要頭像 | 播放該章前，每次只載一章 |
| `codex:<id>` | 單筆圖鑑插圖 | 開啟該筆詳情前 |
| `audio:<key>` | 單一 BGM | 維持 AudioManager 現行按曲目 fetch 與記憶體快取 |

URL manifest 繼續由 Vite glob 產生（`src/art.ts`），不需要把素材搬進 `public/`。

**載入責任定案（2026-07-07 修訂，與已完成的 M5-8b 實作一致）**：採「**各場景 `preload()` 分散載入**」——每個場景在自己的 `preload()` 只載該場景缺少的素材，載入前一律用 `this.textures.exists(key)` 防重抓。M5-8b 三刀已依此路線完成並通過驗證。**不做集中式 AssetManager／`ensureAssets()` 單一入口**（原始構想，已被分散載入取代）；後續段落禁止把已完成的場景 preload 重構成集中制。

### 3-2. 載入流程（依 M5-8b 完成後的實況）

```text
BootScene: boot-core（標題背景＋程式生成貼圖）
  → TitleScene 可操作
  → 選新局／讀檔
  → WorldMapScene.preload()：世界地圖正式圖＋當前旗艦世界／方向圖
  → PortScene.preload()：當前港文化圈底圖＋建築＋主角行走圖＋旗艦
  → StoryScene.preload()：當章專屬背景（每次一章）
  → ShipyardScene／BattleScene／InfoScene.preload()：各自的船圖／裝備／圖鑑單張
```

M5-8d 要補的是：把 Phaser Loader 既有的 `progress`／`loaderror` 事件接到**共用的載入進度元件**（各場景 preload 期間顯示進度、失敗時提供重試與回標題），不是改變上述載入路徑。轉場期間不得留下可重複點擊的舊按鈕。

### 3-3. 超取樣品質

在建立 `Phaser.Game` 前計算品質：

| 模式 | SS | 預設條件 |
|---|---:|---|
| high | 2.0 | 桌面或使用者手動指定 |
| balanced | 1.5 | 平板、一般行動裝置 |
| performance | 1.0 | 小螢幕、低記憶體／低核心數或使用者指定 |

`auto` 只作保守起點，不能宣稱精準辨識 GPU。設定頁新增品質選項並清楚標示「變更後重新載入生效」；選擇存 localStorage，不動遊戲存檔版本。文字 `TEXT_RES` 先維持 4，實測若記憶體過高再獨立調整。

### 3-4. 行動版容器

- viewport 加 `viewport-fit=cover`。
- `html/body/#game` 使用 `100dvh`，並提供 `100vh` fallback。
- 禁止頁面 overscroll、選取與雙指縮放誤觸；canvas 設定 `touch-action: none`。
- 遊戲容器納入 `env(safe-area-inset-*)`。
- 手機直向不把 16:9 遊戲硬拉伸；顯示「建議旋轉為橫向」覆蓋提示，仍保留使用者繼續的選項。

### M5-8c：行動觸控阻斷修正（最高優先，2026-07-07 老闆實機回報）

問題等級：**阻斷級**。iPad／手機實機目前無法穩定移動，也無法完成進港；在此修正通過前，LoadingScene、畫面微調與效能優化不得排在前面。

成因診斷：

- `WorldMapScene` 航向只讀方向鍵／WASD，進港只讀 Enter，沒有純觸控等價操作。
- `PortScene` 雖有點擊移動，但缺少明確的行動方向控制與進入設施按鈕；全場景 pointer handler 也需要避免點 UI 時誤設移動目標。
- 網頁容器未宣告 `touch-action: none`／overscroll 防護，iOS／Android 瀏覽器可能把拖曳判為頁面手勢而取消遊戲輸入。

實作要求：

- 建立世界地圖與港町共用的行動觸控控制層：四方向控制＋情境動作按鈕；桌面鍵盤／滑鼠操作保持不變。
- 世界地圖觸控方向必須能持續航行；靠近港口時動作按鈕顯示「進港」，並與 Enter 共用同一個 action handler。
- 港町保留點地面移動、點建築自動進入，另提供四方向與「進入」按鈕作可靠備援；點 HUD／觸控按鈕不得觸發地面移動。
- `index.html` 補 `touch-action: none`、overscroll／選取防護與 `viewport-fit=cover`，先確保事件交給 Phaser。
- 控制按鈕以手機橫向縮放後仍接近 44 CSS px 為最低設計基準，不能只擴 hitbox 而讓圖示不可辨識。

阻斷驗收：

- [ ] iPad Safari、iPhone Safari、Android Chrome 不接鍵盤，能從世界地圖連續航行、靠近港口並按「進港」進入。
- [ ] 港町可點地面移動、點建築自動進入，也可用方向控制移動並按「進入」。
- [ ] 觸控選單、圖鑑、設定等 UI 不會同時帶動船或人物。
- [ ] 桌面方向鍵／WASD／Enter 與滑鼠操作無回歸。
- [ ] 旋轉或瀏覽器工具列改變尺寸後，控制層仍留在可視安全區。

### M5-8d：載入進度與可恢復錯誤（2026-07-07 修訂範圍）

> **不重構載入路徑**：M5-8b 的「各場景 preload 分散載入」已驗證通過，本段只在其上補使用者可見的進度與錯誤處理。

- 新增共用載入進度元件（建議 `src/loadingHud.ts` 之類單一模組）：任一場景 `preload()` 有待載檔案時，掛 Phaser Loader 的 `progress`／`fileprogress` 事件顯示進度條與「載入中…」文字；載入完成自動移除。用 `BASE_W`／`BASE_H` 排版（見 §八）。
- 失敗處理：掛 `loaderror` 事件——記錄失敗的 key，preload 結束後若有失敗，顯示「部分內容載入失敗」＋【重試】（重新 `scene.restart()` 同參數）＋【回標題】兩個選項；既有 `textures.exists()` fallback 防線保持，不讓場景黑屏或崩潰。
- 驗證方式：DevTools 把某張圖設為 block（或暫時改壞一個 URL）模擬 404，確認重試與返回可用、console 無未處理錯誤；恢復後正常。
- 完成本段後補做 M5-8b 留下的互動式場景回歸（進港／劇情／圖鑑／造船／海戰逐一實測）。

### M5-8d-2：圖檔傳輸量壓縮（總體規劃書 §七-3 原定項目）

> lazy load 解決「不必要的下載」，本段解決「必要下載仍太大」——行動網路下單場景群組的傳輸時間。

- 量測目前單場景群組傳輸量：世界地圖正式圖（`full_map_v3.png` 約 12 MB）、港町單一主題底圖（約 4 MB／張）、單章劇情背景（約 1.4 MB／張）。
- 壓縮手段（擇一或併用，先做投報率最高的大圖）：①PNG 轉有損 WebP（品質 80 左右，Phaser 直接支援）②尺寸本來就超過顯示需求的圖先縮圖再壓。用 Python/Pillow 或 `sharp` 寫進 `tools/`，輸出到新檔名資料夾、保留原圖，`art.ts` glob 指向壓縮版。
- **鐵則**：材質 key 不變（只換 URL 來源）；壓縮前後遊戲內逐場景目視比對，觀感明顯劣化的圖退回原檔；`assets/CREDITS.md` 不需改（同一素材的技術轉檔）。
- 驗收：世界地圖＋單一港町群組合計傳輸量顯著下降（目標至少 −50%），Fast 4G 模擬下「標題→世界地圖可玩」時間記錄前後對比。

## 五、裝置驗收矩陣

| 類別 | 代表視窗／裝置 | 方向 | 必驗項目 |
|---|---|---|---|
| Desktop | Chrome／Edge 1280×720、1366×768、1920×1080 | 橫向 | 畫面置中、鍵鼠、首次載入、60/30 FPS 基線 |
| Chromebook | 1366×768 低階 GPU | 橫向 | performance／balanced、30 FPS 下限、記憶體 |
| iPad | Safari 10.2～11 吋 | 橫向＋直向 | safe area、觸控、旋轉、背景恢復 |
| iPhone 大螢幕 | Safari 390×844 等級 | 橫向＋直向 | 瀏海、工具列、直向提示、文字可讀性 |
| iPhone 小螢幕 | Safari 375×667 等級 | 橫向＋直向 | 最小觸控目標、設定與海戰操作 |
| Android | Chrome 360×800／800×360 | 橫向＋直向 | touch、音訊解鎖、返回／重載、效能 |

桌面瀏覽器裝置模擬先做回歸；iOS Safari 的記憶體、音訊、localStorage 與背景恢復必須由真機補驗，不能只用模擬器宣稱通過。

## 六、驗收門檻

- [ ] Fast 4G 模擬下，標題可操作時間有量測紀錄，且初始下載不再包含全部章節背景、圖鑑與港町底圖。
- [ ] 行動網路首次進入可玩世界目標小於 2 分鐘。
- [ ] 每個延遲載入場景都有進度；失敗可重試或返回。
- [ ] 上述裝置矩陣的核心觸控流程通過，尤其「世界地圖航行 → 進港 → 港町移動／進設施」可全程不接鍵盤完成，console 無未處理錯誤。
- [ ] 行動裝置至少維持 30 FPS；若 high 不達標，auto 能降至 balanced／performance。
- [ ] 旋轉、瀏海、iOS 動態工具列不遮住必要操作。
- [ ] 世界地圖＋單一港町群組傳輸量較壓縮前下降至少 50%，且遊戲內目視無明顯劣化（M5-8d-2）。
- [ ] 現有 v19 存檔可讀；本階段不新增存檔遷移。

## 八、技術約定與踩雷區（執行本規劃前必讀）

> 依據 `memory.md` 既有技術約定與 `2026-07-05_開發工作規範_防閉環迭代機制.md`。以下每條都是已踩過雷或已定案的架構，**違反任何一條＝破壞已驗證的工作**。

1. **改 SS（M5-8f）只動兩處**：`main.ts` 的 `const SS = 2` 改為「建 `Phaser.Game` 前從 localStorage 讀品質設定計算」＋設定頁加選項。**嚴禁**改動：①`applySupersample`／READY hook（各場景 `setZoom(SS)`＋`setOrigin(0,0)` 與「相機接管跟隨」機制——`origin=0` 是 HUD 固定與捲動場景正常的關鍵）②任何場景排版（一律 `BASE_W`／`BASE_H`，禁用 `this.scale.width/height`）③`showModal`／`toast` 等共用 UI 的 BASE 置中。SS=1 或 1.5 時同一套 hook 直接生效，不需要逐場景改。
2. **嚴禁動 `mipmapFilter`**：2026-06-25 踩雷——加了 `LINEAR_MIPMAP_LINEAR` 反而讓文字抓到預先模糊的縮圖更糊，已移除。render 設定只留 `antialias: true`。
3. **`TEXT_RES` 維持 4**：WP-4 老闆親眼比對後定案。若 M5-8f 實測記憶體過高需下調，先提案問老闆，不得自行改。
4. **本機測試固定 `http://127.0.0.1:5173/`**（dev script 有 `--strictPort`）：localStorage 存檔依 origin 分裂，換 port／換 origin 看到的是不同存檔；不得覆蓋未知存檔格，測試用第 9、10 格或全新格。正式站（github.io）也是獨立 origin，本機存檔不會跟過去——這是預期行為，不是 bug。
5. **材質 key 不變原則**：延遲載入與壓縮都只改「URL 來源與載入時機」，`art.ts` 的 key 命名（`m5tb_*`、`shipwd_*`、`storybgch_*`、`codexill_*` 等）與場景取用方式不變；缺材質時的程式生成 fallback 不得移除（回退策略依賴它）。
6. **每段落必跑驗證**（改載入／場景程式後）：`node tools/validate-lazy-assets.mjs`、`node tools/validate-touch-controls.mjs`、`node tools/validate-ui-consistency.mjs`、`node tools/validate-port-layouts.mjs`、`npm run build`；動到資料另跑 `validate-story-data.mjs`／`validate-mates-data.mjs`。全綠才 commit，一段落一 commit＋log（防閉環規範第一條）。
7. **preview 分頁節流**：背景分頁 RAF 暫停，自動化驗證看不到畫面更新時，用 `window.__game.loop.step(t, 16)` 手動推幀，不要誤判為程式壞掉。絕不可把 `Math.random` 換成固定值（Phaser 內部用它產材質 UUID，會崩潰）。
8. **卡住兩次就停**：同一問題修兩輪仍失敗，停下來把現象寫進 log 並回報，不做第三輪盲試（防閉環規範）；session 額度將盡先把工作區 commit 成 WIP。
9. **本階段不動**：存檔結構（維持 v19）、玩法數值、歷史內容文字、`mates.json`／`story` 資料、M5-7 教學（老闆指示暫緩）。
